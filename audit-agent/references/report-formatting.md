# Report Formatting

## Report Structure

The final report MUST follow this structure exactly:

---

```
╔══════════════════════════════════════════════════════════════╗
║              AUDIT REPORT — [Protocol Name]                  ║
║              Generated by Audit Agent v1.0.0                 ║
╚══════════════════════════════════════════════════════════════╝

Protocol Type:  [lending | dex | perps | cdp | bridge | vault | staking | options | nft | oracle | generic]
Files Scanned:  [N] contracts, [NSLOC] nSLOC
Agents Run:     4 parallel agents + 1 validator
Findings:       [N Critical] [N High] [N Medium] [N Low] [N Info]
```

---

## Executive Summary

2–4 sentences describing: what the protocol does, the primary risk surface identified, and the most severe finding in plain English.

---

## Findings

### Critical Findings

#### [C-N] Title

**Severity:** Critical  
**Type:** `bug_class`  
**Contract:** `ContractName`  
**Function:** `functionName`  
**Agents:** [agent-1, agent-3]  

**Impact:** Who loses what. Quantify if possible (e.g., "attacker can drain the entire lending pool").

**Attacker Model:** EOA / malicious contract / flashloan / any user

**Attack Path:**
1. Pre-state: ...
2. Attacker calls `Contract.function(params)`
3. ...
N. Post-state: [profit/loss]

**Affected Code:** `src/Contract.sol:functionName`

**Fix Direction:** 1 paragraph describing the fix. Be specific (e.g., "Move the `accrue()` call to before the state mutation on line X").

---

### High Findings

#### [H-N] Title

[Same format as Critical]

### Medium Findings

#### [M-N] Title

[Same format]

## Appendix: Unvalidated Leads (HYPOTHESIS)

List items that need further validation but couldn't be confirmed:

| # | Pattern | Gate Failed | Next Step |
|---|---|---|---|
| H-1 | ... | Gate C | Read function X |

---

## Appendix: Audit Scope

List all scanned files with line counts.
